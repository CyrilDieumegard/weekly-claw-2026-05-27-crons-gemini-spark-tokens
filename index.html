Résumé

OpenClaw devient utile quand il travaille sans attendre le chat.

Cette édition a surtout montré comment passer d’un assistant qui répond à un agent qui exécute des routines: crons, agents dédiés, modèles locaux, scraping, livraisons Telegram ou Discord.

CronsAgentsLM StudioApifyTokens
Idée clé: automatiser les tâches répétitives, mais garder le contrôle sur le modèle utilisé et le canal de sortie.
Point clé 1

Les cron jobs: programmer du travail récurrent

Un cron job OpenClaw permet de lancer une tâche à heure fixe: tous les matins, chaque semaine, une seule fois, ou selon un rythme précis.

QuoiDécrire la tâche: météo, veille, résumé, audit, scraping, backup ou cockpit projet.
QuandChoisir la fréquence: matin, soir, heure précise, hebdomadaire ou date unique.
OùEnvoyer le résultat dans le dashboard, Telegram, Discord ou une session dédiée.
Un bon cron économise de l’attention. S’il ajoute du bruit, il faut le supprimer ou le resserrer.
Point clé 2

Cas d’usage concrets sortis pendant la session

  • Brief quotidien: météo, marchés, actualité, devises.
  • Résumé email du soir, directement envoyé sur Telegram.
  • Cockpit projets: ce qui est en cours, bloqué, à faire.
  • Backup quotidien des données importantes.
  • Leçons récurrentes: piano, italien, Python.
  • Transformation automatique de vidéos longues en shorts.
Le déclic: OpenClaw peut devenir un opérateur de routines, pas juste une fenêtre de discussion.
Point clé 3

Scraping et veille: très puissant, mais à cadrer

Gilles a montré un exemple simple: scraper immobilier.ch pour les locations de 3 pièces à Genève, puis construire un fichier avec localité, pièces, prix, balcon et lien d’annonce.

Ce qui marcheUn prompt simple peut suffire pour extraire des données structurées si le site laisse passer.
Ce qui bloqueBeaucoup de sites protègent leurs données ou interdisent le scraping.
Option avancéeApify peut gérer des scrapers spécialisés, par exemple LinkedIn ou veille concurrentielle.
Bon usageRemonter uniquement les annonces, profils ou signaux qui valent une action.
Point clé 4

Agents: pas besoin d’en créer 50

La discussion a été claire: multiplier les agents pour le spectacle crée souvent plus de confusion que de valeur.

Agent principalPour les tâches importantes, le code, les décisions complexes et les échanges quotidiens.
Agent localPour les routines simples qui peuvent tourner via LM Studio sans coût de tokens cloud.
Agent spécialiséPour un rôle net: scraping, veille, reporting, contenu ou support.
Créer un agent seulement s’il simplifie le routage, les outils, le modèle ou les coûts.
Point clé 5

Le local sert surtout à maîtriser les coûts

Un cron qui tourne tous les jours avec un gros modèle cloud peut vite coûter cher. Pour les tâches simples, un modèle local suffit souvent.

Cloud : décisions difficiles, code complexe, qualité maximale
Local : routines, brouillons, idées, résumés simples, veille récurrente
Routage : le bon modèle pour la bonne tâche, pas le modèle le plus cher par défaut
Le bon modèle n’est pas toujours le plus fort. C’est celui qui fait le job au meilleur coût.
Point clé 6

LM Studio: attention au préchargement

Quand un modèle local est appelé pour la première fois, il peut être lent parce qu’il doit être chargé en mémoire. Une fois préchargé, les réponses deviennent plus fluides.

À retenir

La bonne architecture

Un agent principal pour le travail exigeant. Un agent local pour les routines. Des cron jobs pour les tâches répétitives. Des canaux propres pour recevoir seulement ce qui compte.

Le futur proche: moins de prompts manuels, plus de routines intelligentes, mais avec une vraie discipline sur les coûts de tokens.
